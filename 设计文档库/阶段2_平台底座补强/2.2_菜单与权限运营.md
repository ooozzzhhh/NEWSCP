# 2.2 菜单与权限运营设计文档

> 版本：v1.0 | 日期：2026-04-16 | 所属阶段：阶段2 - 平台与安全底座补强  
> **与《重构计划.md》4.2 节「2.2 菜单与权限运营（P0）」标题一致。** 实现状态：**已完成**。

## 1. 功能概述

**目标：** 在阶段 1「角色分配权限」基础上，提供**系统级权限树**的在线维护能力：新增/编辑/删除菜单与按钮节点、维护排序与路由元数据、启用/停用；与角色授权（`PUT /api/sys/roles/{id}/permissions`）共用同一张 `nscp_sys_permission` 表。

**与 1.2 的差异：** 1.2 侧重 RBAC 分配关系；本项侧重**资源树本身的运营**（原种子数据可由实施/管理员在界面演进，无需改 SQL 发版）。

**依赖：** 角色与权限（1.2）、菜单动态权限（1.4）中的权限模型说明。

---

## 2. 数据模型

沿用 `nscp_sys_permission`（见 [1.2 角色与权限](../阶段1_系统基础/1.2_角色与权限.md) §2.2）。要点：

- `perm_type`：`MENU` | `BUTTON`
- `perm_code`：全局唯一，小写字母、数字、`:`、`_`、`-`，长度 3～128
- `parent_id`：`0` 表示根；**按钮不可挂在根下**，必须挂在 `MENU` 下
- 运营树接口使用 `selectAllIncludeDisabled`，含 `DISABLED` 节点，供管理端编辑

---

## 3. 后端接口设计

**Controller：** `com.newscp.backend.sys.permission.PermissionOperateController`  
**基础路径：** `/api/sys/permissions`

| 方法 | 路径 | 所需权限 | 功能 |
|------|------|----------|------|
| GET | `/api/sys/permissions/operate-tree` | `sys:perm:list` | 全量权限树（含停用），用于运营页 |
| POST | `/api/sys/permissions` | `sys:perm:create` | 新增节点 |
| PUT | `/api/sys/permissions/{id}` | `sys:perm:update` | 更新节点（名称、路由、排序、父节点、状态等） |
| DELETE | `/api/sys/permissions/{id}` | `sys:perm:delete` | 逻辑删除；有子节点或仍被角色引用时不允许删除 |

**业务规则摘要（`PermissionOperateService`）：**

- 删除：存在未删除子节点 → 报错；存在角色绑定 → 报错。
- 修改父节点：不能指向自身、不能落到自己子树下；父不能为 `BUTTON`。

**与只读菜单树区分：**

| 接口 | 用途 |
|------|------|
| `GET /api/sys/permissions/tree` | 登录用户可见菜单树（阶段 1 动态菜单，通常仅 `ENABLED`） |
| `GET /api/sys/permissions/operate-tree` | 权限运营页全树 |

## 4. 前端设计

| 路径 | 说明 |
|------|------|
| `frontend/src/pages/system/permissions/index.tsx` | 权限树运营 UI |
| `frontend/src/api/system.ts` | `fetchPermissionOperateTree`、`createPermission`、`updatePermission`、`deletePermission` |
| `frontend/src/router/routeComponentMap.tsx` | 新增菜单时 `component_path` 需与映射表一致，否则动态路由无法解析 |

新增 `MENU` 时应在 `routeComponentMap` 注册组件键，并在种子或界面中配置 `component_path`、`route_path`，否则用户侧路由无法打开页面。

---

## 5. 角色联动

角色授权仍通过阶段 1 接口维护 `nscp_sys_role_permission`。运营页删除权限前须解除所有角色绑定，避免「角色仍引用已删权限码」导致鉴权与菜单不一致。
