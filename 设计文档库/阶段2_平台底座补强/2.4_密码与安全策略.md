# 2.4 密码与安全策略设计文档

> 版本：v2.0 | 日期：2026-04-16 | 所属阶段：阶段2 - 平台与安全底座补强
> **与《重构计划.md》4.2 节「2.4 密码与安全策略（P1）」标题对齐。**
> **参考来源**：scp-foundation `auth_rbac_password_config` 表及 `PasswordConfigServiceImpl` 实现

## 1. 设计目标

| 目标 | 说明 |
|------|------|
| 策略可配置 | 管理员通过界面配置密码复杂度、有效期、锁定规则等，无需改代码 |
| 登录流程衔接 | 登录时自动检查：锁定状态 → 密码正确性 → 密码过期 → 强制改密 |
| 改密流程衔接 | 改密时校验新密码是否符合当前策略 |
| 租户隔离 | 每个租户独立维护自己的密码策略 |
| 简洁落地 | 不做 Redis 失败计数、不做图形验证码（P2 再考虑），保持最小实现 |

## 3. 数据库设计

### 3.1 密码策略表 `nscp_sys_password_policy`

**设计原则**：每个租户一条记录（1:1），不存在则使用系统默认值。

```sql
CREATE TABLE nscp_sys_password_policy (
  id              BIGINT        NOT NULL AUTO_INCREMENT PRIMARY KEY,
  tenant_id       VARCHAR(64)   NOT NULL COMMENT '租户ID（每租户一条）',

  -- ═══ 密码复杂度 ═══
  min_length      INT           NOT NULL DEFAULT 8    COMMENT '最小长度',
  max_length      INT           NOT NULL DEFAULT 32   COMMENT '最大长度',
  require_digit   TINYINT(1)    NOT NULL DEFAULT 1    COMMENT '要求包含数字',
  require_lower   TINYINT(1)    NOT NULL DEFAULT 1    COMMENT '要求包含小写字母',
  require_upper   TINYINT(1)    NOT NULL DEFAULT 0    COMMENT '要求包含大写字母',
  require_special TINYINT(1)    NOT NULL DEFAULT 0    COMMENT '要求包含特殊符号',

  -- ═══ 密码有效期 ═══
  expire_enabled  TINYINT(1)    NOT NULL DEFAULT 0    COMMENT '是否启用密码有效期',
  expire_days     INT           NOT NULL DEFAULT 90   COMMENT '密码有效天数',
  alert_before_days INT         NOT NULL DEFAULT 7    COMMENT '过期前N天预警（0=不预警）',

  -- ═══ 强制改密 ═══
  force_change_default TINYINT(1) NOT NULL DEFAULT 1  COMMENT '首次使用默认密码时强制修改',
  force_change_on_rule_update TINYINT(1) NOT NULL DEFAULT 0 COMMENT '策略更新后强制不符合用户修改密码',

  -- ═══ 账户锁定 ═══
  lock_enabled    TINYINT(1)    NOT NULL DEFAULT 1    COMMENT '是否启用登录失败锁定',
  lock_threshold  INT           NOT NULL DEFAULT 5    COMMENT '连续失败N次后锁定',
  lock_duration   INT           NOT NULL DEFAULT 30   COMMENT '锁定时长（分钟）',
  auto_unlock     TINYINT(1)    NOT NULL DEFAULT 1    COMMENT '是否到期自动解锁',

  -- ═══ 默认初始密码 ═══
  default_password VARCHAR(128) NOT NULL DEFAULT 'Admin@2026' COMMENT '管理员重置密码时的默认密码（明文存储，登录时BCrypt比对）',

  -- ═══ 审计字段 ═══
  created_by      VARCHAR(64),
  created_at      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_by      VARCHAR(64),
  updated_at      DATETIME      ON UPDATE CURRENT_TIMESTAMP,

  UNIQUE KEY uk_tenant (tenant_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='密码与安全策略（每租户一条）';
```

### 3.2 与用户表的关联

用户表 `nscp_sys_user` 已有以下字段，**无需新增**：

| 字段 | 用途 |
|------|------|
| `status` | `ACTIVE`/`DISABLED`/`LOCKED` |
| `locked_until` | 锁定解锁时间 |
| `login_fail_count` | 连续失败次数 |
| `pwd_changed_at` | 密码最后修改时间（用于有效期判断） |

### 4.2 REST 接口

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/api/sys/password-policy` | `sys:security:view` | 查询当前租户策略（不存在返回默认值） |
| PUT | `/api/sys/password-policy` | `sys:security:edit` | 更新当前租户策略 |

接口少而精：只有查和改，没有创建和删除。首次查询时若无记录，Service 层自动以默认值插入一条。

### 4.3 PasswordPolicyService 核心方法

```java
public class PasswordPolicyService {

    /**
     * 获取当前租户的密码策略（不存在则创建默认值）
     */
    public PasswordPolicy getEffective(String tenantId);

    /**
     * 更新策略，需校验：
     * - minLength >= 6, maxLength <= 64, minLength <= maxLength
     * - 四种复杂度要求至少选一种
     * - lockThreshold >= 1, lockDuration >= 1
     * - defaultPassword 自身必须符合当前复杂度规则
     */
    public void update(String tenantId, PasswordPolicyDTO dto);

    /**
     * 校验密码是否符合策略，返回错误信息列表（空=通过）
     * 被 AuthService.login()、UserController.changePassword() 调用
     */
    public List<String> validatePassword(String password, PasswordPolicy policy);

    /**
     * 检查用户密码是否过期
     * @return null=未过期, String=过期提示信息
     */
    public String checkExpired(SysUser user, PasswordPolicy policy);

    /**
     * 检查是否需要强制改密（默认密码/规则更新后不符合）
     * @return null=不需要, String=强制改密原因
     */
    public String checkForceChange(SysUser user, String rawPassword, PasswordPolicy policy);

    /**
     * 生成过期预警信息（登录成功后检查）
     * @return null=不需要预警, String=预警提示
     */
    public String getExpirationWarning(SysUser user, PasswordPolicy policy);
}
```

### 4.4 密码复杂度校验逻辑

```java
public List<String> validatePassword(String password, PasswordPolicy policy) {
    List<String> errors = new ArrayList<>();

    if (password.length() < policy.getMinLength())
        errors.add("密码长度不能少于" + policy.getMinLength() + "位");
    if (password.length() > policy.getMaxLength())
        errors.add("密码长度不能超过" + policy.getMaxLength() + "位");
    if (policy.isRequireDigit() && !password.chars().anyMatch(Character::isDigit))
        errors.add("需要包含数字");
    if (policy.isRequireLower() && !password.chars().anyMatch(Character::isLowerCase))
        errors.add("需要包含小写字母");
    if (policy.isRequireUpper() && !password.chars().anyMatch(Character::isUpperCase))
        errors.add("需要包含大写字母");
    if (policy.isRequireSpecial() &&
        !password.chars().anyMatch(c -> !Character.isLetterOrDigit(c) && !Character.isWhitespace(c)))
        errors.add("需要包含特殊符号");

    return errors;
}
```

### 4.5 登录流程改造（AuthService.login）

改造后的登录检查顺序：

```
1. 查用户 → 不存在则报错
2. 加载密码策略 → policyService.getEffective(tenantId)
3. 检查用户状态（DISABLED / LOCKED + 自动解锁逻辑）
   - LOCKED 判断改为读取 policy.lockDuration 而非硬编码 30
4. 密码校验
   - 失败：lockThreshold / lockDuration 从策略读取
   - 成功：清零失败计数
5. 【新增】密码过期检查 → policyService.checkExpired()
   - 过期则返回 code=600 + message，前端跳转改密页
6. 【新增】强制改密检查 → policyService.checkForceChange()
   - 默认密码 / 不符合新规则 → 返回 code=600
7. 生成 JWT Token
8. 【新增】过期预警 → policyService.getExpirationWarning()
   - 有预警则写入响应的 pwdExpireWarning 字段
```

**关键约定**：`code=600` 为密码策略拦截专用错误码，前端据此展示强制改密对话框。

### 4.6 改密流程改造（UserController.changePassword）

```
1. 校验旧密码
2. 加载策略 → policyService.getEffective(tenantId)
3. 校验新密码 → policyService.validatePassword(newPassword, policy)
   - 不通过则返回所有错误信息
4. BCrypt 加密 → 更新 password_hash 和 pwd_changed_at
```

### 4.7 重置密码流程（UserController.resetPassword）

```
1. 管理员操作：将目标用户密码重置为策略中的 default_password
2. 更新 pwd_changed_at = null（标记为"未改过密码"，配合 force_change_default 使用）
```

### 5.2 密码强度预览算法

```typescript
function calcStrength(policy: PasswordPolicy): 'weak' | 'medium' | 'strong' {
  let score = 0;
  if (policy.minLength > 12) score += 40;
  else if (policy.minLength > 8) score += 20;

  const typeCount = [policy.requireDigit, policy.requireLower,
                     policy.requireUpper, policy.requireSpecial]
                    .filter(Boolean).length;
  if (typeCount >= 4) score += 40;
  else if (typeCount >= 3) score += 30;
  else if (typeCount >= 2) score += 15;

  score += typeCount * 5;

  return score >= 80 ? 'strong' : score >= 41 ? 'medium' : 'weak';
}
```

### 5.3 登录页强制改密对话框

当登录接口返回 `code=600` 时：

1. 弹出改密对话框（覆盖登录页面）
2. 输入旧密码（预填登录时输入的密码）、新密码、确认密码
3. 实时展示新密码是否符合策略（前端本地校验 + 提交时后端二次校验）
4. 修改成功后自动重新登录

### 5.4 登录后过期预警

当登录响应中包含 `pwdExpireWarning` 字段时：

- 在页面顶部或右上角展示一条可关闭的提示条
- 内容示例："您的密码将在 5 天后过期，请及时修改"
- 附带"立即修改"按钮

## 6. 与 scp-foundation 的差异说明

| 维度 | scp-foundation | NEWSCP（本设计） | 原因 |
|------|---------------|-----------------|------|
| 字段命名 | YES/NO 字符串 | TINYINT(1) 布尔值 | 新项目约定：布尔用 TINYINT |
| 密码加密传输 | AES 加密 | HTTPS 明文传输 | 新项目全站 HTTPS，无需前端加密 |
| 失败计数存储 | Redis | 用户表 `login_fail_count` 字段 | 单机部署，无需 Redis；已有字段复用 |
| 验证码 | 完整实现 | 不做（P2） | 优先级低，先保证核心策略可用 |
| 管理员/普通用户锁定阈值 | 分开配置 | 统一一个阈值 | 简化，实际需求不大 |
| 租户关系 | 全局一条配置 | 每租户一条 | NEWSCP 是 SaaS，租户需独立策略 |
