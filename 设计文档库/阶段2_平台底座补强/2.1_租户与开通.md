# 2.1 租户与开通设计文档

> 版本：v1.0 | 日期：2026-04-16 | 所属阶段：阶段2 - 平台与安全底座补强  
> **与《重构计划.md》4.2 节「2.1 租户与开通（P0）」标题一致。** 实现状态：**已完成**。

## 1. 功能概述

**目标：** 在阶段 1 单租户用户模型之上，补齐多租户开通与访问控制：租户主数据维护、用户与多租户绑定、登录时校验租户可用性与用户开通范围；支持已登录用户通过请求头切换「运行时租户」（在权限允许范围内）。

**依赖：** 用户管理（1.1）、角色与权限（1.2）、登录认证（1.5）。

**后续依赖本模块：** 各业务域在数据访问层使用 `TenantContext` 做租户隔离。

## 2. 数据库设计

### 2.1 nscp_sys_tenant（租户表）

表结构与初始化脚本见 `backend/src/main/resources/db/stage2_init.sql`。核心字段：

| 字段 | 说明 |
|------|------|
| tenant_id | 租户业务标识，唯一，创建后不可改（业务层约束） |
| tenant_name | 展示名称，可改 |
| status | `ENABLED` / `DISABLED` |
| expire_at | 可选，到期后登录拒绝 |
| deleted | 逻辑删除（`@TableLogic`） |

### 2.2 nscp_sys_user_tenant（用户-租户关联）

| 字段 | 说明 |
|------|------|
| user_id, tenant_id | 联合主键 |
| is_default | 同一用户多租户时的默认租户偏好（创建/展示用） |

### 2.3 平台种子与特殊租户

- 超级管理员租户 `tenant_id = admin`：禁止删除（`TenantService` 硬编码保护）。
- 种子角色：`SUPER_ADMIN_ROLE`、`TENANT_ADMIN_ROLE`（见 `stage2_init.sql`），与租户运营、权限运营能力绑定。

## 3. 后端接口设计

**Controller：** `com.newscp.backend.tenant.TenantController`  
**基础路径：** `/api/sys/tenants`

| 方法 | 路径 | 所需权限 | 功能 |
|------|------|----------|------|
| GET | `/api/sys/tenants` | `sys:tenant:list` | 分页查询（keyword 匹配 tenantId/tenantName，可按 status 筛选） |
| POST | `/api/sys/tenants` | `sys:tenant:create` | 创建租户并可选批量绑定用户、指定默认用户 |
| GET | `/api/sys/tenants/{id}` | `sys:tenant:read` | 详情 |
| PUT | `/api/sys/tenants/{id}` | `sys:tenant:update` | 更新基本信息并可选替换成员列表 |
| DELETE | `/api/sys/tenants/{id}` | `sys:tenant:delete` | 逻辑删除；`admin` 租户不可删；存在绑定用户时不可删 |
| GET | `/api/sys/tenants/{id}/users` | `sys:tenant:read` | 租户已绑定用户列表 |
| PUT | `/api/sys/tenants/{id}/users` | `sys:tenant:assign-user` | 仅更新成员与默认用户（与 update 内 `replaceTenantUsers` 同逻辑） |

**租户 ID 校验（创建）：** 正则 `^[A-Za-z0-9_-]{3,64}$`（见 `TenantService.normalizeTenantId`）。

**默认用户规则：** 若传入 `defaultUserId`，必须属于 `userIds`；未传时，`admin` 租户且成员含超管用户 id=1 则默认 id=1，否则取列表首条。

## 4. 登录与请求链路

### 4.1 登录侧校验（`AuthService`）

在密码校验前完成：

1. 租户存在、未删除、`status=ENABLED`。
2. `expire_at` 未早于当前时间。
3. `nscp_sys_user_tenant` 存在该用户与该租户绑定。
4. 用户已分配至少一个角色（跨租户角色码查询）。

未登录默认租户：请求体未带 `tenantId` 时使用 `admin`（常量 `DEFAULT_TENANT_ID`）。

### 4.2 运行时租户切换（`JwtAuthenticationFilter`）

- 请求头：`X-Tenant-Id`（与 JWT 内 `tenantId` 区分）。
- 若请求头与 Token 内租户一致：直接使用。
- 若不一致：**超级管理员**（角色含 `TenantContext.SUPER_ADMIN_ROLE`）可在头为任意**已启用**租户时切换。
- **普通用户**：头中租户须在 `nscp_sys_user_tenant` 绑定且租户启用，否则回退为 Token 内租户。
- 解析结果写入 `TenantContext`，供后续数据访问使用。
