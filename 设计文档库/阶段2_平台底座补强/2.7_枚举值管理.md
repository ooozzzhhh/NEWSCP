# 2.7 枚举值管理设计文档

> 版本：v1.0 | 日期：2026-04-16 | 所属阶段：阶段2 - 平台与安全底座补强 + 阶段3 - 系统配置中心
> **与《重构计划.md》「2.7 前端枚举值映射治理」及「阶段3 系统配置中心」对齐。**
> **参考来源**：scp-foundation `custom_enums` 表 + `EnumsCache` + `AbstractCacheService`

---

## 1. 设计目标与原则

| 目标 | 说明 |
|------|------|
| 可靠 | 枚举数据持久化到数据库，有审计字段，有唯一约束防重复 |
| 先进 | 内置枚举自动注册 + 运行时可扩展；TypeScript 类型安全；前端缓存减少请求 |
| 便捷 | 管理员在界面增删改枚举值；开发者在代码中定义枚举自动同步到数据库；前端一行代码消费 |
| 不需要太大变更 | 表结构简洁通用；类型/值分两表（经典 dict_type + dict_value 模式）；接口设计面向长期稳定 |

### 与 scp-foundation 的核心差异

| 维度 | scp-foundation | NEWSCP（本设计） | 原因 |
|------|---------------|-----------------|------|
| 表结构 | 单表 `custom_enums`（type+value 混存） | 双表 `dict_type` + `dict_item`（类型与值分离） | 分离后查询更高效，类型级别可独立管理排序/启用 |
| 缓存 | 自研 HashMap + ReentrantReadWriteLock | Spring Cache（Caffeine 本地缓存） | 利用 Spring 生态，代码更少，TTL 自动过期 |
| 内置枚举同步 | 启动时全量删后插 | 启动时增量 upsert（有则更新，无则插入） | 避免 ID 变动、避免用户自定义的 sort_order 等被冲掉 |
| 前端消费 | 每个页面单独调接口 | 全局批量加载 + React Context 提供 | 减少重复请求，统一消费入口 |
| 租户隔离 | 无 tenant_id | 有 tenant_id | SaaS 必须 |

## 2. 数据库设计

### 2.1 枚举类型表 `nscp_sys_dict_type`

```sql
CREATE TABLE nscp_sys_dict_type (

  editable    TINYINT(1)    NOT NULL DEFAULT 1 COMMENT '是否允许用户编辑枚举值（BUILTIN 类型可设为0）',
  status      TINYINT(1)    NOT NULL DEFAULT 1 COMMENT '1=启用, 0=禁用',
  sort_order  INT           NOT NULL DEFAULT 0 COMMENT '排序序号',

  created_by  VARCHAR(64),
  created_at  DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_by  VARCHAR(64),
  updated_at  DATETIME      ON UPDATE CURRENT_TIMESTAMP,
  deleted     TINYINT(1)    NOT NULL DEFAULT 0,

  UNIQUE KEY uk_tenant_code (tenant_id, type_code, deleted)

```

### 2.2 枚举值表 `nscp_sys_dict_item`

```sql
CREATE TABLE nscp_sys_dict_item (
  id           BIGINT        NOT NULL AUTO_INCREMENT PRIMARY KEY,
  tenant_id    VARCHAR(64)   NOT NULL COMMENT '租户ID',
  type_code    VARCHAR(64)   NOT NULL COMMENT '所属类型编码（关联 dict_type.type_code）',
  value        VARCHAR(64)   NOT NULL COMMENT '枚举值（接口传输用，如 APPROVED）',
  label        VARCHAR(128)  NOT NULL COMMENT '显示文本（如 已审核）',
  label_en     VARCHAR(128)  COMMENT '英文显示文本（国际化预留）',
  color        VARCHAR(32)   COMMENT '前端展示颜色/Badge样式（如 success, warning, #FF0000）',
  extra        VARCHAR(512)  COMMENT '扩展 JSON（预留，如图标、CSS类名等）',
  sort_order   INT           NOT NULL DEFAULT 0 COMMENT '排序序号',
  status       TINYINT(1)    NOT NULL DEFAULT 1 COMMENT '1=启用, 0=禁用',
  is_default   TINYINT(1)    NOT NULL DEFAULT 0 COMMENT '是否默认选中项',
  remark       VARCHAR(500)  COMMENT '备注',
  created_by   VARCHAR(64),
  created_at   DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_by   VARCHAR(64),
  updated_at   DATETIME      ON UPDATE CURRENT_TIMESTAMP,
  deleted      TINYINT(1)    NOT NULL DEFAULT 0,

  UNIQUE KEY uk_tenant_type_value (tenant_id, type_code, value, deleted),
  INDEX idx_type_code (tenant_id, type_code, status, sort_order)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='枚举/字典项';
```

### 2.3 设计说明

**为什么用 `type_code` 关联而不是 `type_id` 外键？**
- 代码中引用枚举时用 `"ORDER_STATUS"` 这种字符串常量更直观
- 内置枚举同步时不依赖自增 ID，幂等性更好
- 前端请求 `/dict/items?typeCodes=ORDER_STATUS,PRODUCT_TYPE` 更易理解

**`color` 字段的作用**：
- 前端表格/Badge 展示状态时直接读取，无需硬编码颜色映射
- 值为 shadcn Badge variant 名（`default`/`success`/`warning`/`destructive`）或十六进制色值
- 不填则前端使用默认样式

**`extra` 字段的作用**：
- JSON 格式，为未来扩展预留（图标、前端样式类名、业务标记等）
- 当前阶段可不使用

### 3.3 DictService 核心逻辑

### 3.4 缓存策略

使用 Spring Cache + Caffeine（本地缓存），不引入 Redis：

```java
@Configuration
@EnableCaching
public class CacheConfig {
    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager manager = new CaffeineCacheManager("dictDropdown");
        manager.setCaffeine(Caffeine.newBuilder()
            .maximumSize(500)        // 最多 500 个缓存条目
            .expireAfterWrite(Duration.ofMinutes(10))  // 10分钟过期
        );
        return manager;
    }
}
```

缓存 Key：`tenantId + ":" + typeCodes排序拼接`

缓存失效：DictItem/DictType 的增删改操作后，调用 `cacheManager.getCache("dictDropdown").clear()` 清除该租户缓存。

### 3.5 内置枚举自动注册机制

#### 3.5.1 定义 DictEnum 接口

```java
/**
 * 业务枚举实现此接口即可自动注册到字典表。
 * 开发者只需写 Java 枚举 + 实现接口，启动时自动同步。
 */
public interface DictEnum {
    /** 枚举类型编码（类级别，对应 dict_type.type_code） */
    String typeCode();
    /** 枚举类型名称 */
    String typeName();
    /** 枚举值 */
    String value();
    /** 显示文本 */
    String label();
    /** 排序（可选） */
    default int sortOrder() { return 0; }
    /** 颜色（可选） */
    default String color() { return null; }
}
```

#### 3.5.2 业务枚举示例

```java
public enum OrderStatus implements DictEnum {
    DRAFT("DRAFT", "草稿", "default", 1),
    SUBMITTED("SUBMITTED", "已提交", "warning", 2),
    APPROVED("APPROVED", "已审核", "success", 3),
    REJECTED("REJECTED", "已驳回", "destructive", 4);

    private final String val;
    private final String lbl;
    private final String clr;
    private final int sort;

    @Override public String typeCode() { return "ORDER_STATUS"; }
    @Override public String typeName() { return "订单状态"; }
    @Override public String value()    { return val; }
    @Override public String label()    { return lbl; }
    @Override public int sortOrder()   { return sort; }
    @Override public String color()    { return clr; }
}
```

#### 3.5.3 启动同步流程

```java
@Component
public class DictEnumRegistrar implements ApplicationRunner {

    @Override
    public void run(ApplicationArguments args) {
        // 1. 扫描所有实现 DictEnum 接口的枚举类（通过 Spring 的 ClassPathScanningCandidateComponentProvider）
        // 2. 按 typeCode 分组
        // 3. 对每个 typeCode：
        //    - dict_type: INSERT ... ON DUPLICATE KEY UPDATE type_name=VALUES(type_name)
        //    - dict_item: INSERT ... ON DUPLICATE KEY UPDATE label=VALUES(label)
        //      （只更新 label，不覆盖 color/sort_order/extra 等用户可能已手动调整的字段）
        // 4. 清除缓存
    }
}
```

**增量 upsert 的好处**（相比 scp-foundation 的全删后插）：
- 用户在界面上修改的 color、sort_order 不会被启动冲掉
- dict_item 的 ID 稳定，不会因重启而变化
- 新增枚举值自动出现，删除的枚举值保留为历史（可手动禁用）

### 4.2 前端统一消费方案

#### 4.2.1 核心 Hook：`useDict`

```typescript
// src/hooks/useDict.ts

/**
 * 核心消费 Hook：传入 typeCode 数组，返回响应式的枚举映射。
 * 自动批量请求 + 全局缓存（同一 typeCode 不会重复请求）。
 */
export function useDict(...typeCodes: string[]): DictMap;

// 返回类型
type DictMap = Record<string, DictOption[]>;
type DictOption = {
  value: string;
  label: string;
  color?: string;
  isDefault?: boolean;
};
```

#### 4.2.2 使用示例

```tsx
// 页面中使用
function OrderListPage() {
  const dict = useDict('ORDER_STATUS', 'PRODUCT_TYPE');

  return (
    <Select options={dict.ORDER_STATUS} />
    // 或在表格列中渲染
    <Badge variant={getLabelAndColor(dict.ORDER_STATUS, row.status).color}>
      {getLabelAndColor(dict.ORDER_STATUS, row.status).label}
    </Badge>
  );
}
```

#### 4.2.3 工具函数

### 4.3 前端枚举映射治理规则（与重构计划 2.7 对齐）

| 规则 | 实现方式 |
|------|---------|
| 统一映射出口 | 所有枚举通过 `useDict()` Hook 获取，禁止页面内硬编码 |
| 内部值命名规范 | 使用稳定语义值（`ACTIVE`、`DRAFT`），禁止 `__ALL__` 等占位命名 |
| 接口层只传内部值 | API 请求/响应只用 `value`，展示层通过 `getDictLabel()` 翻译 |
| 后端字典优先 + 前端兜底 | `useDict()` 从后端加载；网络异常时 fallback 到 `src/constants/dict-fallback.ts` |

### 5.2 与密码策略的关系

密码策略使用独立表 `nscp_sys_password_policy`（见 2.4 文档），不放入 `nscp_sys_param`。原因：
- 密码策略字段多且有类型约束（INT/BOOLEAN），放 param 表需反复序列化
- 密码策略需整体读取（一次读所有字段），独立表性能更好
- 系统参数表适合 KV 型简单配置

## 7. 长期稳定性分析

| 变更场景 | 需要做什么 | 影响范围 |
|----------|-----------|---------|
| 新增业务枚举（如新模块） | 写 Java Enum 实现 `DictEnum` → 重启自动注册 | 零改动 |
| 管理员增加新的枚举值 | 界面操作 | 零改动 |
| 需要国际化 | dict_item 已有 `label_en`，扩展多语言列即可 | 加列 + 前端 i18n 适配 |
| 需要枚举值关联业务规则 | 使用 `extra` JSON 字段存储规则参数 | 零表结构改动 |
| 从 MyBatis-Plus 迁移到 Hibernate | Entity 加 JPA 注解，Mapper 改 Repository | 接口不变 |
| 多租户枚举隔离 | 已内置 `tenant_id`，天然支持 | 零改动 |
