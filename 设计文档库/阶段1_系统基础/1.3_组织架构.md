# 1.3 组织架构设计文档

> 版本：v1.0 | 日期：2026-04-14 | 所属阶段：阶段1 - 系统基础

## 1. 功能概述

实现租户内的部门树形结构管理，支持：
- 部门增删改查（树形展示 + 列表展示）
- 用户与部门的关联（一个用户有一个主部门）
- 「直属上级」关系维护（审批流"直属上级审批"规则的数据基础）

**依赖：** 用户管理（1.1）  
**被依赖：** 审批流（阶段5）——"直属上级"/"部门负责人"审批节点需要组织树数据

## 2. 数据库设计

### 2.1 nscp_sys_dept（部门表）

```sql
CREATE TABLE nscp_sys_dept (
    id          BIGINT          NOT NULL AUTO_INCREMENT PRIMARY KEY COMMENT '主键',
    tenant_id   VARCHAR(64)     NOT NULL                           COMMENT '租户ID',
    dept_name   VARCHAR(128)    NOT NULL                           COMMENT '部门名称',
    parent_id   BIGINT          NOT NULL DEFAULT 0                 COMMENT '父部门ID（0=根节点）',
    leader_id   BIGINT                                             COMMENT '部门负责人用户ID（关联 nscp_sys_user.id）',
    sort_order  INT             NOT NULL DEFAULT 0                 COMMENT '同级排序（升序）',
    remark      VARCHAR(256)                                       COMMENT '备注',
    created_by  VARCHAR(64)     NOT NULL,
    created_at  DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_by  VARCHAR(64),
    updated_at  DATETIME                 ON UPDATE CURRENT_TIMESTAMP,
    deleted     TINYINT(1)      NOT NULL DEFAULT 0,

    INDEX idx_tenant_parent (tenant_id, parent_id),
    INDEX idx_tenant_deleted (tenant_id, deleted)
) COMMENT '系统部门表';
```

**字段说明：**

| 字段 | 说明 |
|------|------|
| parent_id | 0 表示根节点（每个租户的顶级部门）；通过递归查询构建树 |
| leader_id | 部门负责人，NULL 表示未设置；用于审批流中"部门负责人审批"规则 |

### 2.2 nscp_sys_user_dept（用户-部门关联表）

```sql
CREATE TABLE nscp_sys_user_dept (
    user_id     BIGINT          NOT NULL COMMENT '用户ID',
    dept_id     BIGINT          NOT NULL COMMENT '部门ID',
    is_primary  TINYINT(1)      NOT NULL DEFAULT 1 COMMENT '是否主部门（1=是，目前一个用户只有一个部门）',

    PRIMARY KEY (user_id, dept_id),
    INDEX idx_dept_id (dept_id)
) COMMENT '用户-部门关联表';
```

> **当前设计：** 一个用户只允许属于一个主部门（`is_primary` 保留扩展性，但业务逻辑限制 is_primary=1 的记录只能有一条）。

### 2.3 直属上级关系说明

直属上级关系**不单独建表**，通过以下方式推导：
1. 查询用户所在部门（nscp_sys_user_dept）
2. 找该部门的 leader_id（部门负责人）
3. 如 leader_id == 当前用户自己，则向上找父部门的 leader_id

若需要明确指定某用户的直属上级（绕过部门层级），可在 nscp_sys_user 表后期增加 `supervisor_id` 字段（当前阶段暂不加，审批流阶段评估）。

### 2.4 初始数据示例

```sql
-- demo-tenant 的根部门
INSERT INTO nscp_sys_dept (tenant_id, dept_name, parent_id, sort_order, created_by, created_at)
VALUES ('demo-tenant', 'NEWSCP公司', 0, 0, 'system', NOW());

-- 子部门示例
INSERT INTO nscp_sys_dept (tenant_id, dept_name, parent_id, sort_order, created_by, created_at)
VALUES
  ('demo-tenant', '供应链计划部', 1, 10, 'system', NOW()),
  ('demo-tenant', '技术研发部',   1, 20, 'system', NOW());
```

## 3. 后端接口设计

### 3.1 接口列表

| 方法 | 路径 | 所需权限 | 功能 |
|------|------|---------|------|
| GET | `/api/sys/depts/tree` | `sys:dept:list` | 获取当前租户的完整部门树 |
| GET | `/api/sys/depts` | `sys:dept:list` | 获取部门列表（平铺，用于下拉选择） |
| POST | `/api/sys/depts` | `sys:dept:create` | 新增部门 |
| PUT | `/api/sys/depts/{id}` | `sys:dept:update` | 修改部门（名称/负责人/排序/父部门） |
| DELETE | `/api/sys/depts/{id}` | `sys:dept:delete` | 删除部门（有子部门或关联用户时拒绝） |
| GET | `/api/sys/depts/{id}/users` | `sys:dept:list` | 查看部门下的用户列表 |

### 3.2 关键请求/响应格式

#### 获取部门树

**请求：** `GET /api/sys/depts/tree`

**响应：**
```json
{
  "code": 0,
  "data": [
    {
      "id": 1,
      "deptName": "NEWSCP公司",
      "parentId": 0,
      "leaderId": null,
      "leaderName": null,
      "sortOrder": 0,
      "userCount": 10,
      "children": [
        {
          "id": 2,
          "deptName": "供应链计划部",
          "parentId": 1,
          "leaderId": 5,
          "leaderName": "李四",
          "sortOrder": 10,
          "userCount": 5,
          "children": []
        }
      ]
    }
  ]
}
```

#### 新增部门

```json
POST /api/sys/depts
{
  "deptName": "需求计划组",
  "parentId": 2,
  "leaderId": 10,
  "sortOrder": 10,
  "remark": "负责需求预测与计划"
}
```

**业务规则：**
- parentId 必须存在且属于当前租户
- 同一父部门下，dept_name 不允许重复

#### 修改部门

```json
PUT /api/sys/depts/{id}
{
  "deptName": "需求计划组（更名）",
  "leaderId": 12,
  "sortOrder": 5,
  "parentId": 2
}
```

**业务规则：**
- 不允许将自身设为自己的父节点或子孙节点（防循环）
- 修改 parentId（移动部门）时校验目标父节点不是当前节点的子孙节点

#### 删除部门

**业务规则：**
- 有子部门时拒绝，提示「存在子部门，请先删除子部门」
- 有关联用户时拒绝，提示「部门下存在 N 个用户，请先将用户迁移至其他部门」

**左右布局（桌面端）：**

```
┌─────────────────────────────────────────────────────────┐
│  [新增根部门 按钮]                                          │
├─────────────┬───────────────────────────────────────────┤
│  部门树      │  选中部门的详情面板                           │
│  ─ NEWSCP  │  部门名称：供应链计划部                        │
│    ├ 供应链  │  负责人：李四                                 │

│    │  └ 库存 │                                            │
```

| 字段 | 组件 | 说明 |
|------|------|------|

## 5. 业务逻辑说明

### 5.1 部门树与租户隔离
- MP 租户插件自动在 SQL 注入 `WHERE tenant_id = ?`，无需手写
- `buildTree()` 操作的数据集已经是当前租户数据，不存在跨租户问题

### 5.2 部门负责人（leader_id）的用途
| 使用场景 | 说明 |
|---------|------|
| 审批流（阶段5） | 审批节点类型「部门负责人」：取发起人所在部门的 leader_id |
| 审批流（阶段5） | 审批节点类型「直属上级」：查找上级部门的 leader_id（递归直到找到非本人） |
| 组织展示 | 部门详情页展示负责人姓名 |

### 5.3 用户-部门关联的维护时机
- 用户新增时：同步写入 nscp_sys_user_dept（deptId 来自用户创建表单）
- 用户修改时：如果 deptId 变化，先删旧关联再插新关联
- 部门删除时：前置检查 nscp_sys_user_dept 中是否有关联用户，有则拒绝删除

## 6. 与 scp-foundation 的主要差异

| 对比点 | scp-foundation（参考） | NEWSCP（本设计） |
|--------|----------------------|----------------|
| 根部门 | tenant_root=YES 标记根部门 | parent_id=0 标识根节点（更简洁） |
| 部门-用户 | auth_rbac_dept_user | nscp_sys_user_dept |
| 直属上级 | 无明确设计 | 通过 leader_id 推导；后期可补 supervisor_id |
| 部门移动 | 无循环引用检测 | 修改 parentId 前校验目标不是自身子孙节点 |
| 用户多部门 | 支持（auth_rbac_dept_user 无唯一约束） | 仅支持单主部门（当前阶段简化） |
