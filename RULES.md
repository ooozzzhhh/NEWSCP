# NEWSCP RULES

更新时间：2026-04-15  
适用范围：`NEWSCP/` 全仓库（前端、后端、数据库、文档）

## 1. 技术栈固定（不反复讨论）

- 前端：React + Vite + TypeScript
- UI：shadcn/ui + Tailwind CSS（已有页面改造时需兼容现有 Dark Neon 视觉）
- 请求与状态：Axios + TanStack Query（渐进引入）
- 后端：Spring Boot 3 + Hibernate（Spring Data JPA）
- 数据库：MySQL
- 接口返回统一：`{ code, data, msg }`

## 2. 数据库与命名规范（硬约束）

- 业务表统一前缀：`nscp_`
- 字段统一下划线命名
- 业务表必须包含公共字段：`id tenant_id created_by created_at updated_by updated_at deleted`
- 高并发/高频查询必须考虑 `tenant_id` 联合索引
- 多租户采用“共享库共享表 + tenant_id 行级隔离”

## 3. 多租户与角色模型（硬约束）

- 超级管理员租户固定为：`admin`
- 角色分层：
  - `SUPER_ADMIN_ROLE`：可管理所有租户与全功能
  - `TENANT_ADMIN_ROLE`：仅管理所属租户下用户与功能菜单
- 默认要求：
  - 非超管请求必须受租户隔离
  - 超管跨租户操作必须显式指定目标租户（例如请求头 `X-Tenant-Id`）

## 4. 交付顺序（阶段依赖）

- 阶段 0～12 以 `重构计划.md` 第四章为唯一权威
- 阶段 1（系统基础）完成后才能进入阶段 2/3
- 系统配置中心（阶段 3）先于业务模块落地
- 通知先于审批；主数据先于业务配置与核心业务数据

## 5. 前端视觉规范（Dark Neon）

- 全站统一 Dark Neon，不允许页面私有配色漂移
- 所有颜色走全局 token 风格，不回退白底工业风
- 输入框、按钮、表格、分页、弹窗保持统一组件外观
- 页面需兼容桌面与移动端基础可用性

## 6. 视觉新增红线（补充）

- 主操作按钮禁止彩虹/多段渐变，统一使用单色主按钮（青色系）
- 分页器中的下拉框必须主题化（深色背景 + 浅色文字 + 焦点描边）
- 勾选框状态色使用绿色（已确认）
- 筛选区控件统一高度：`input/select/ghost button = 36px`（禁止页面内自行偏离）
- Select 弹层不得遮挡触发框：默认需与触发框保持可见间距（不允许“覆盖式弹出”）
- 统一枚举值规范：内部值（value）与显示值（label）必须分离管理；禁止使用 `__ALL__` 这类临时占位值进入用户界面
- 映射管理位置统一：前端枚举映射集中在 `src/constants`（或等价统一目录），页面只引用，不重复硬编码

## 7. 前端组件库与复用规则（必看）

- 组件库主目录：`frontend/src/components/`
- shadcn 基础组件目录：`frontend/src/components/ui/`
- 当前已落地可复用组件（截至 2026-04-15）：
  - `frontend/src/components/ui/button.tsx`
  - `frontend/src/components/ui/checkbox.tsx`
  - `frontend/src/components/ui/select.tsx`
  - `frontend/src/components/ui/pagination.tsx`
  - 业务封装：`frontend/src/components/PageSizeSelect.tsx`
- 复用优先级：
  1. 优先复用 `components/ui/*`（shadcn 基础组件）
  2. 再复用 `components/*` 里的业务封装
  3. 最后才允许新建组件（禁止页面内重复造同类控件）
- 样式约束：复用组件时必须遵循 Dark Neon 和第 6 节的统一控件约定（高度、下拉弹层、勾选色等）
- 新增组件流程：
  1. 先检查 `components/ui` 与 `components` 是否已有等价能力
  2. 无等价能力再新增，并补充到本节“已落地可复用组件”清单
  3. 涉及全局行为（如下拉、分页、表单控件）需同步更新规则说明

## 8. AI 开发工作流（执行要求）

- 每次功能开发遵循：
  1. 先读旧逻辑/旧表结构
  2. 先定 DDL 再写后端
  3. 后端接口完成后再写前端页面
  4. 联调通过后再进入下一个功能点
- 一次只做一个可运行子目标，避免大批量无验证提交
- 变更后必须至少执行对应构建/测试命令并记录结果

## 9. 变更管理与质量门槛

- 新增规则优先写入本文件，再改代码
- 涉及架构、租户安全、权限边界的改动，必须先与本规则对齐
- 若与 `重构计划.md` 冲突，以双方最新一致版本为准；冲突项需先更新文档再开发

## 10. 设计文档库同步规则（强制）

- 设计文档库路径：`设计文档库/`
- 每次“一个可运行子目标”开发完成后，必须同步更新设计文档，不得只改代码不改文档
- 至少更新以下内容：
  1. 实现状态：`已完成 / 进行中 / 未开始`
  2. 实现方式：核心技术方案、关键约束、与计划差异
  3. 代码落点：关键文件路径（后端/前端/SQL）
  4. 待办项：当前未完成边界与下一步
- 更新顺序要求：
  1. 先更新 `重构计划.md` 的阶段状态（若有变更）
  2. 再更新 `设计文档库/README.md` 索引状态
  3. 最后更新对应阶段文档（如 `设计文档库/阶段2_平台底座补强/README.md` 或子文档）
- 提交前检查项：若本次改动涉及功能行为变化，PR/提交说明中必须明确“文档已同步”。
