# NEWSCP 从零开始开发计划

> 版本：v1.3 | 更新时间：2026-04-13
> 背景：1 人 + AI，1～2 个月，验证 AI 开发路线可行性
> 原则：以最小时间成本做出可用产品；旧产品只用于参考业务逻辑，不做对接

---

## 一、技术选型（定死，不要在这上面浪费时间）

| 层 | 选择 | 理由 |
|----|------|------|
| 前端框架 | React + Vite + TypeScript | shadcn 官方基础，生态最全 |
| UI 组件 | shadcn/ui + Tailwind CSS | AI 生成代码质量最高的组件库 |
| 状态/请求 | TanStack Query + Axios | 服务端状态管理，少写很多同步代码 |
| 表单 | React Hook Form + Zod | 类型安全，AI 生成代码稳定 |
| 路由 | React Router v6 | 标准 |
| 图表 | Recharts | shadcn 官方推荐，与 Tailwind 一致 |
| 后端框架 | Spring Boot 3 + MyBatis-Plus | 与现有 Java 技术栈一致 |
| 数据库 | MySQL（沿用） | 表结构重新设计 |
| 接口规范 | RESTful，统一响应 `{ code, data, msg }` | — |

## 二、数据库设计原则（开发前定好，改起来最贵）

### 2.1 表设计模式

**所有业务表必须有的公共字段：**
```sql
id          BIGINT PRIMARY KEY AUTO_INCREMENT
tenant_id   VARCHAR(64)   -- 租户隔离（多租户场景）
created_by  VARCHAR(64)
created_at  DATETIME
updated_by  VARCHAR(64)
updated_at  DATETIME
deleted     TINYINT(1) DEFAULT 0  -- 逻辑删除
```

**头行表结构（适用于单据类：订单、预测版本等）：**
```
xxx_header  -- 单据头（ID、状态、时间范围、备注等）
xxx_line    -- 明细行（关联 header_id，存产品/数量等）
```

**扩展属性表（适用于字段不固定的场景，谨慎使用）：**
```
xxx_attr（entity_id, attr_key, attr_value, attr_type）
```
> 注：不是所有表都要用 EAV。简单配置表用宽表没问题；只有字段真的无法预知时才用 attr 扩展。

### 2.2 命名规范

- 表前缀统一用 `nscp_`（new scp）
- 格式：`nscp_<领域>_<实体>`，例如 `nscp_ord_customer`、`nscp_sto_safety_config`
- 字段用下划线命名法

## 五、AI 开发工作方式

### 5.1 每个功能域的标准流程

```
1. 先读旧代码（旧表结构 + 旧前端页面逻辑）→ 整理给 AI 做上下文
2. 让 AI 设计新表 DDL → 你审核确认
3. 让 AI 写后端接口（Controller + Service + Mapper）
4. 让 AI 写前端页面（从 shadcn 组件拼装）
5. 联调 → 修 bug → 完成
```

### 5.2 减少返工的关键约定

- **第 0 步定好的技术选型不再变动**，后续不讨论"要不要换 xxx 库"
- **每个功能域开始前**，先把旧产品对应的表结构和页面截图给 AI 看，再让它设计新的
- **接口返回格式**统一，不要一个接口 `{ data: [] }` 一个接口 `{ list: [] }`
- **类型定义（TypeScript）优先**：先写类型，再写组件，AI 生成的代码会更准确
- **不要让 AI 一次写太多**：一次一个页面或一个接口，跑通后再做下一个

### 5.3 用旧产品节省时间的正确姿势

| 借鉴内容 | 怎么用 |
|----------|--------|
| 业务字段定义 | 读旧表结构，理解每个字段含义，设计新表时参考 |
| 计算逻辑 | 读旧 Java 代码，理解规则，让 AI 用新方式实现 |
| 页面交互 | 打开旧前端截图，作为 AI 生成新页面的参考 |
| 接口参数 | 读旧前端的 API 调用，理解入参出参，设计新接口时参考 |

## 六、主要风险（只列真实风险）

| 风险 | 应对 |
|------|------|
| 数据库设计早期出错，后期改表成本高 | 第 2 步前把公共字段规范定死；表结构改动在数据量小时（开发期）代价低 |
| AI 生成的代码结构不一致，积累后难维护 | 做完第 1 个功能后提取一个"标准页面模板"，后续都让 AI 照着模板写 |
| 需求范围蔓延，60 天不够 | 严格按上面顺序推进；分析看板和协同功能如果时间不够可以砍掉 |
| shadcn 定制样式耗时 | 第 0 步定好颜色/字体变量，之后不再改全局样式 |
| 多租户隔离漏写，导致数据泄露 | MyBatis-Plus 租户插件自动注入 tenant_id，不依赖开发者手写 WHERE |

### 9.1 站内通知 + 私信

**目标定位（工业软件合格线）**

| 功能 | 是否实现 | 说明 |
|------|----------|------|
| 通知 Bell + 未读数 | ✅ | 顶栏常驻，实时更新 |
| 系统通知列表 | ✅ | 审批状态变化、系统预警、任务指派 |
| 已读/未读标记 | ✅ | 支持一键全部已读 |
| 基础私信（一对一） | ✅ | 发送/接收/历史记录 |
| 消息推送（实时） | ✅ | WebSocket，页面不刷新也能收到 |
| 群组消息 / @功能 / 已读回执 | ❌ | 超出合格线，不做 |

**数据表设计**

```sql
-- 通知表（系统事件推送）
nscp_sys_notification
  id, tenant_id, receiver_id, type(APPROVAL/ALERT/SYSTEM),
  title, content, related_type, related_id,  -- 关联到具体业务单据
  is_read(0/1), created_at

-- 私信表
nscp_msg_message
  id, tenant_id, sender_id, receiver_id,
  content, is_read(0/1), created_at

-- 私信会话表（聚合两人之间的对话）
nscp_msg_conversation
  id, tenant_id, user_a_id, user_b_id,
  last_message_id, unread_count, updated_at
```

**技术实现**
- 后端：Spring WebSocket（`@EnableWebSocketMessageBroker`）
- 前端：SockJS + STOMP Client，建立连接后订阅个人频道 `/user/{userId}/notification`
- 非实时场景（页面刷新时）：普通 REST 接口拉取未读数和列表

**开发工作量估算：约 4～5 天**

### 9.2 审批流 / 工作流

**参考设计**：`产品设计/产品需求/IPS/IPS_审批流能力建设/审批流设计大纲.md`（已有完整设计大纲，含角色体系、功能清单、状态机、与业务模块集成契约）

**使用 Flowable，加一层业务封装**

```
业务代码 → WorkflowService（封装层）→ Flowable API
```

**新产品实现范围（按优先级）**

P0（必须做，核心可用）：
- 流程模板管理（表单配置式，非 BPMN 拖拽设计器）：模板列表、基本信息、串行审批节点配置、审批人规则（固定人员/指定角色/发起人自选）
- 审批待办中心：待办列表、已办列表、未读标记

- 审批发起：从业务单据触发、预览审批链
- 进度追踪：我的申请列表、审批节点时间线、历史记录
- 状态变更通知（通过/拒绝/退回 → 推送给申请人；新任务 → 推送给审批人）

P1（核心稳定后迭代）：
- 转交、加签
- 条件分支（按表单字段值自动路由）
- 撤回（申请人主动撤回）、催办
- 批量审批

P2（暂不做）：
- 模板版本管理
- 模拟测试
- 监控大盘
- IM 消息卡片（钉钉/企业微信集成）

**关键技术决策（参考大纲第十节待决策点）**

| 问题 | 新产品决策 |
|------|-----------|
| 流程模板设计器 | 表单配置式（参考飞书风格），不做 BPMN 可视化设计器 |
| 底层引擎 | 保留 Flowable |
| IM 集成 | 暂不做，站内通知覆盖 |
| 退回方式 | 先只做"退回到发起人"，P1 再支持退回到指定节点 |

| 跨租户模板 | 租户独立配置，不共享 |

**前置依赖（必须先完成）**：用户管理 + 组织架构（审批人规则"直属上级"/"部门负责人"需要组织树）、通知系统

### 9.3 低代码图表

**设计参考**：融合旧产品「动态透视宽表框架」和「需求优先级字段配置化」两份设计文档的核心思想，在新产品中用更简洁的方式实现。

**核心设计原则：VIEW 白名单 + 配置驱动查询 + 用户无需写 SQL**

```
业务表（随业务迭代变化）
    ↓ 开发人员维护
语义 VIEW（稳定接口层，每个业务域一个或多个）
    ↓ 注册到白名单
nscp_chart_datasource（VIEW 注册表，安全白名单）
    ↓ 挂载维度/指标/筛选字段定义
配置驱动查询引擎（按白名单字段生成安全 SQL）
    ↓
图表渲染
```

参考旧产品 `rpt_table_registry` 的白名单思想：只有注册过的 VIEW 才能被图表引用，后端生成查询时表名和列名均做白名单校验，彻底消除注入风险。

**用户操作路径（无需写 SQL）**

```
选数据主题（如"需求预测"）
  → 选维度（时间/产品/库存点）       ← 来自 nscp_chart_dimension 白名单
  → 选指标（预测数量/达成率）         ← 来自 nscp_chart_metric 白名单
  → 选筛选条件（时间范围/产品类型）    ← 来自 nscp_chart_filter 白名单
  → 选图表类型（折线/柱状/饼图/数据表）
  → 系统生成安全查询 → 渲染图表
  → 保存 → 拖拽组合成看板
```

**数据表设计（参考旧产品 rpt_* 系列表结构）**

```sql
-- VIEW 注册表（安全白名单，开发人员维护）
-- 参考：rpt_table_registry
nscp_chart_datasource
  id, tenant_id, code, name,
  view_name(对应数据库VIEW名，保存时校验格式只允许字母数字下划线),
  description, module_code(所属模块), status, sort_order

-- 维度字段池（行区域，用户选择"按什么分组"）
-- 参考：rpt_dimension_field
nscp_chart_dimension
  id, datasource_id, col_name(VIEW中已有的列名),
  display_name, dim_type(TIME/TEXT/ENUM/NUMBER),
  enum_class(有枚举时框架自动翻译), is_default, sort_order, is_enabled

-- 指标字段池（值区域，对哪个数值列做什么聚合）
-- 参考：rpt_measure_field
nscp_chart_metric
  id, datasource_id, col_name, display_name,
  agg_func(SUM/COUNT/AVG/MAX/MIN), unit, sort_order, is_enabled

-- 筛选字段池（筛选区域）
-- 参考：rpt_filter_field
nscp_chart_filter
  id, datasource_id, col_name, display_name,
```

**VIEW 命名规范与开发契约**

每完成一个业务域，同步创建对应的语义 VIEW，并注册到 `nscp_chart_datasource`：

```sql
-- 示例：需求预测数据主题 VIEW
CREATE VIEW v_chart_demand_forecast AS
SELECT
  h.tenant_id,
  DATE_FORMAT(h.plan_date, '%Y-%m')  AS time_month,
  p.product_code,
  p.product_name,
  sp.stock_point_code,
  sp.stock_point_name,
  l.forecast_qty,
  h.version_code
FROM nscp_forecast_header h
  JOIN nscp_forecast_line l    ON l.header_id = h.id AND l.deleted = 0
  JOIN nscp_product p          ON p.id = l.product_id AND p.deleted = 0
  JOIN nscp_stock_point sp     ON sp.id = l.stock_point_id AND sp.deleted = 0
WHERE h.deleted = 0;
-- 注意：VIEW 中不加 tenant_id 过滤，由后端租户插件统一注入
```

**后端安全查询生成（参考旧产品聚合引擎 SQL 生成逻辑）**

```
1. 用户请求带：datasource_code + 选中维度列表 + 选中指标列表 + 筛选值
2. 后端从 nscp_chart_dimension / nscp_chart_metric 加载白名单
3. 校验用户选中的每个字段 ∈ 白名单（不在白名单直接拒绝）
4. 从 nscp_chart_datasource 取 view_name（再次校验格式）
5. 生成 SQL：
   SELECT [白名单维度列], [AGG_FUNC(白名单指标列)] FROM [已注册VIEW]
   WHERE tenant_id = #{tenantId}  ← MyBatis-Plus 租户插件注入
   [AND 用户筛选值，全部 #{} 参数化]
   GROUP BY [白名单维度列]
   LIMIT 10000
6. 执行，返回图表数据
```

关键安全保障：表名来自白名单表（不信任用户输入），列名来自白名单表（不信任用户输入），筛选值全部参数化（不拼字符串）。

**与旧产品设计的主要简化点**

| 旧产品（动态透视宽表框架） | 新产品 |
|--------------------------|--------|
| 双引擎（聚合引擎 + 指标树引擎） | 单引擎（聚合类），供需平衡类暂不做 |
| 聚合引擎直接拼 UNION ALL + 动态建表 | VIEW 作为数据源，取数逻辑收敛到 VIEW 里 |
| SPI DataProvider + FormulaRegistry | VIEW 内部处理数据逻辑，不需要 SPI |
| rpt_table_registry（物理表注册） | nscp_chart_datasource（VIEW 注册，更安全） |
| 支持实施顾问配置+扫描物理表 | 由开发人员维护 VIEW 注册，更简单 |

**开发工作量估算：约 8～10 天**（图表配置引擎 + 看板拖拽布局）

### 9.4 配置驱动设计原则

> 新产品的核心设计思想：**用户能在系统界面完成绝大部分配置，不需要修改代码**

| 层级 | 配置驱动体现 |
|------|-------------|
| 系统层 | 枚举/字典值、系统参数、通知模板 → 管理员在界面配置 |
| 用户权限层 | 用户、角色、权限 → 管理员在界面配置 |
| 组织层 | 部门、上下级关系 → 管理员在界面配置（供审批流"直属上级"规则使用） |
| 审批层 | 审批模板、审批节点、审批人规则 → 管理员在界面配置（Flowable 模板由系统动态生成） |
| 图表层 | 数据主题、维度、指标、图表、看板 → 用户在界面配置 |
| 业务配置层 | 安全库存计算规则、优先级因子权重、净需求规则 → 用户在界面配置 |
| 业务数据层 | 订单、预测等核心数据的录入和维护 → 用户操作 |

**实现原则**：
- 枚举值（状态、类型、分类）存数据库，不写死代码，管理员可增删改
- 业务规则（计算方式、权重、阈值）存配置表，不写死计算逻辑里
- 审批规则（谁审、几级）存 Flowable 配置，不写死流程定义
- 只有图表 VIEW 的注册需要开发介入（因为 VIEW 本身是 SQL，不可避免）

### 8.1 隔离模式选择

三种常见模式对比：

| 模式 | 说明 | 适合场景 |
|------|------|----------|
| 独立数据库 | 每个租户一个 DB | 数据安全要求极高，成本高 |
| 共享 DB + 独立 Schema | 每个租户一个 Schema | 中等隔离，MySQL 支持不好 |
| **共享 DB + 行级隔离（推荐）** | 所有租户同表，靠 tenant_id 区分 | SaaS 早期标准做法，成本低，扩展方便 |

**决策：使用行级隔离（shared database, shared schema）**。未来数据量大或有客户要求独立时，再升级到独立数据库模式。

---

### 8.2 数据库层：每张表都有 tenant_id

```sql
-- 所有业务表公共字段（必须，不可省略）
id          BIGINT          NOT NULL AUTO_INCREMENT PRIMARY KEY,
tenant_id   VARCHAR(64)     NOT NULL COMMENT '租户ID',
created_by  VARCHAR(64)     NOT NULL,
created_at  DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
updated_by  VARCHAR(64),
updated_at  DATETIME        ON UPDATE CURRENT_TIMESTAMP,
deleted     TINYINT(1)      NOT NULL DEFAULT 0 COMMENT '逻辑删除'
```

**联合索引**：高频查询字段 + tenant_id 必须建联合索引，例如：
```sql
INDEX idx_tenant_product (tenant_id, product_code)
```

### 8.3 后端层：MyBatis-Plus 租户插件自动注入

用 MyBatis-Plus 的 `TenantLineInnerInterceptor`，配置一次之后，**所有 SQL 自动加 `WHERE tenant_id = ?`**，不需要每个 Mapper 手写。

```java
// 配置类（只需写一次）
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor() {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
    interceptor.addInnerInterceptor(new TenantLineInnerInterceptor(new TenantLineHandler() {
        @Override
        public Expression getTenantId() {
            // 从当前请求上下文取 tenant_id（ThreadLocal）
            return new StringValue(TenantContext.getTenantId());
        }
        @Override
        public String getTenantIdColumn() {
            return "tenant_id";
        }
    }));
    return interceptor;
}
```

---

### 8.4 后端层：tenant_id 的传递链路

```
客户端请求
  → 携带 JWT Token（token 里包含 tenant_id）
  → 后端 Filter/Interceptor 解析 JWT
  → 把 tenant_id 存入 ThreadLocal（TenantContext）
  → MyBatis-Plus 插件从 ThreadLocal 取值注入 SQL
  → 请求结束时清除 ThreadLocal（防止内存泄漏）
```

---

### 8.5 前端层：租户信息的处理

- 登录接口返回 JWT，前端存在 `localStorage`（或 `sessionStorage`）
- 每次请求通过 Axios 拦截器自动在 Header 加 `Authorization: Bearer <token>`
- **前端不需要手动传 tenant_id**，后端从 token 里解析

### 8.7 注意事项

- **tenant_id 用 UUID 字符串，不用自增 ID**：防止枚举攻击（攻击者猜测其它租户的 ID）
- **超级管理员接口要绕过租户过滤**：MyBatis-Plus 插件支持配置哪些表不过滤，系统表（如 `nscp_sys_tenant`）要排除
- **不同租户的数据量差异大时**：单张表数据量大了可以按 tenant_id 分区（后期优化，现在不用考虑）
